# AI 리포트 초안

이 문서는 공식 DOCX 양식에 옮기기 위한 작업 초안입니다. 저장소 기준 구현 상태와 일치하는 내용만 정리합니다.

## 기본 정보

- 팀명 또는 이름: `[최종 입력 예정]`
- 프로젝트명: `VIVA`

## 1. 기획

### 사용자와 문제

VIVA의 1차 사용자는 교사, 2차 사용자는 학생, 3차 사용자는 교육 운영자입니다.

현재 교육 현장에서는 과제, 보고서, 프로젝트 결과물이 좋아 보여도 학생이 실제로 이해했는지는 별도로 확인하기 어렵습니다. 생성형 AI가 보편화되면서 `결과물의 완성도`와 `학생의 실제 이해`가 더 쉽게 분리되고 있습니다.

교사는 모든 학생에게 실시간 구술 검증을 하기 어렵고, 학생은 AI를 사용하더라도 최종적으로 자신의 이해를 증명할 기회가 필요합니다. 운영자는 수업 단위에서 어떤 개념에서 반복적으로 이해가 무너지는지 보고 싶어 합니다.

VIVA는 이 문제를 `학습`이 아니라 `학습 검증`의 문제로 정의합니다.

### 핵심 기능

VIVA는 제출 이후 검증 레이어로 동작합니다.

1. 교사가 과제, 루브릭, 핵심 개념, 위험 신호, 제출물을 입력합니다.
2. AI가 학생 제출물 기반으로 왜형, 전이형, 반례형 질문 3개를 생성합니다.
3. 학생은 텍스트 또는 음성(STT)으로 답변합니다.
4. 시스템은 제출물, 루브릭, 질문, 학생 답변을 함께 비교합니다.
5. 분석 결과를 `이해 충분`, `표면 암기`, `제출물 의존`, `핵심 오개념`, `불확실` 구조로 정리합니다.
6. 교사는 근거를 검토하고 최종 판단을 저장합니다.
7. 운영자는 반복 오개념, 누락 개념, 최근 세션을 요약 화면에서 확인합니다.

### 기대 효과

- 교사는 모든 학생에게 긴 구술 시험을 하지 않아도 이해 검증 흐름을 만들 수 있습니다.
- 학생은 AI를 사용했더라도 최종적으로 자신의 이해를 설명해야 합니다.
- 운영자는 수업 단위의 오개념 패턴과 취약 개념을 집계해 수업 개선에 활용할 수 있습니다.

## 2. AI 사용 역할

### AI를 왜 썼는가

이 프로젝트의 AI는 일반 챗봇이 아니라 `질문 생성 엔진`, `이해 분석 엔진`, `근거 구조화 엔진` 역할을 맡습니다.

- 질문 생성 단계: 제출물과 루브릭을 바탕으로 학생별 질문 세트를 만듭니다.
- 분석 단계: 제출물, 질문, 답변, 루브릭을 함께 비교해 이해 분류와 근거를 구조화합니다.
- 운영 단계: 결과는 교사가 빠르게 읽을 수 있는 형식으로 정리됩니다.

### 사용 도구와 모델

- 문서/기획 보조: AI 기반 문서 정리와 구조 설계
- 구현 보조: 코드 구조화, 스키마 설계, 테스트 보조
- 런타임 AI: OpenAI API 기반 질문 생성/답변 분석
- 저장 구조: Neon(Postgres) 또는 로컬 파일 저장
- 배포: Vercel

현재 배포 환경에서는 OpenAI 크레딧 문제로 프로덕션 런타임이 mock fallback을 사용할 수 있습니다. 이 상태는 코드상으로 fallback reason이 드러나도록 처리되어 있습니다.

### 역할 분리

- 강한 모델 하나로 모든 단계를 처리하지 않고, 질문 생성과 분석을 분리해 설계했습니다.
- 출력은 자유 텍스트가 아니라 JSON schema 검증을 거칩니다.
- AI는 최종 판정자가 아니며, 교사가 마지막 결정을 내립니다.

## 3. 데이터 흐름

1. 교사 화면에서 과제와 루브릭을 입력합니다.
2. `POST /api/questions`가 질문 세트를 생성하고 검증 세션을 저장합니다.
3. 학생은 `/student/[verificationId]` 링크로 진입합니다.
4. 학생 답변은 텍스트 또는 브라우저 STT로 수집됩니다.
5. `POST /api/analyze`가 세션 기반 분석을 수행하고 분석 결과를 저장합니다.
6. 교사는 `/teacher` 또는 `/teacher/verifications/[verificationId]`에서 결과를 검토합니다.
7. `POST /api/teacher-decisions`가 교사 판단을 저장합니다.
8. 운영자는 `/operator`와 `/api/summary`로 패턴을 확인합니다.
9. `GET /api/export`로 CSV/JSON export를 수행합니다.

## 4. 비용, 효율, 재현성

- mock fallback 경로를 두어 실AI 장애나 미설정 상황에서도 흐름을 검증할 수 있습니다.
- 질문 생성과 분석은 구조화 응답을 사용해 재현성과 유지보수성을 높였습니다.
- 저장은 Neon과 로컬 파일 어댑터를 분리해 개발과 배포 환경을 함께 지원합니다.
- `lint`, `unit test`, `build`, `smoke:http`, `smoke:url`로 자동 검증합니다.

## 5. 현재 구현 상태

- 교사, 학생, 운영자, 설정, 로그인/권한 흐름 구현
- STT + 텍스트 fallback 구현
- 학생 링크 잠금/재개방 구현
- 세션 상세, 최근 세션 검색, 재불러오기 구현
- export 및 운영자 요약 구현
- Vercel 배포와 프로덕션 헬스체크 완료

## 6. 한계와 후속 계획

- OpenAI 크레딧 충전 전까지 프로덕션 실AI 품질 검증은 제한됩니다.
- 디자인은 현재 구조 확정 이후 별도 재설계 예정입니다.
- 인증은 현재 역할 기반 간소화 흐름이며, 실제 상용화 단계에서는 사용자 계정 체계와 감사 로그 강화가 필요합니다.
