# AI 빌딩 리포트 초안

이 문서는 공식 DOCX 양식에 옮겨 적기 위한 초안이다. 공개 저장소 문서이므로 연락처와 서명 정보는 넣지 않는다.

## 기본 정보

- 팀명(또는 이름): `[입력 예정]`
- 프로젝트명: `VIVA`

## 1. 기획

### 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

VIVA의 1차 사용자는 교강사이며, 2차 사용자는 학생/수강생, 3차 사용자는 교육 운영자입니다. 현재 교육 현장에서는 과제, 보고서, 프로젝트 결과물이 좋아도 그것이 곧 학생의 실제 이해를 의미하지 않는 문제가 커지고 있습니다. 특히 AI 도구가 보편화되면서 결과물의 완성도와 학습자의 실제 이해 수준이 더 쉽게 분리되고 있습니다.

교강사는 제출물만으로 학생 이해도를 확신하기 어렵고, 모든 학생에게 장시간 개별 구술평가를 진행하기에는 시간과 비용이 부족합니다. 학생은 AI를 활용했더라도 최종적으로 자기 이해가 되었는지 증명할 방법이 필요하며, 운영자는 수료율이나 만족도만이 아니라 실제 학습 품질을 보여줄 근거가 필요합니다.

VIVA는 이러한 문제를 `학습 자체`가 아니라 `학습 검증의 실패`라는 관점에서 해결하려는 서비스입니다.

### 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

VIVA는 제출 이후 검증 레이어로 동작합니다. 교강사가 과제와 루브릭, 핵심 개념을 입력하면 시스템은 학생 제출물을 기반으로 왜형 질문, 전이형 질문, 반례형 질문을 자동 생성합니다. 학생은 짧은 텍스트 또는 음성 답변으로 응답하고, 시스템은 제출물, 루브릭, 질문, 학생 답변을 함께 비교하여 이해 신뢰도와 판단 근거를 구조화합니다.

교강사에게는 결과 분류, 판단 근거 요약, 빠진 개념, 충돌 문장, 재설명 포인트를 제공하고, 교사는 이를 바탕으로 최종 판단을 내릴 수 있습니다. 운영자에게는 반복 오개념과 취약 개념 분포를 보여주는 집계 정보를 제공합니다.

### 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

교강사는 모든 학생에게 장시간 구술평가를 하지 않고도 더 정교한 이해 검증을 수행할 수 있고, 결과물의 품질과 실제 이해 수준을 분리해서 볼 수 있습니다. 학생은 AI 사용 여부 자체를 의심받는 구조 대신 실제로 이해했는지를 설명하고 증명할 기회를 가지게 됩니다. 운영자는 과정 품질을 단순 수료율이 아니라 이해 검증 데이터로 확인할 수 있어 강의 개선과 리메디얼 설계의 근거를 확보할 수 있습니다.

## 2. AI 활용 전략

### 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

이 프로젝트에서 AI는 단순 챗봇이 아니라 `질문 생성기`, `의미 판독기`, `리포트 구조화 엔진`의 역할을 수행합니다. 질문 생성 단계에서는 제출물과 루브릭을 바탕으로 학생별로 다른 질문을 생성해야 하므로 LLM 기반 추론이 필요합니다. 분석 단계에서는 학생 답변이 제출물의 핵심 주장과 얼마나 일치하는지, 루브릭 핵심 개념을 실제로 다루는지, 조건 변경 시 개념 전이가 가능한지, 제출물과 답변 사이에 논리 충돌이 있는지 등을 의미 수준에서 비교해야 하므로 역시 AI 활용이 필수적입니다.

도구와 모델은 실제 구현 단계에서 역할별로 분리해 사용합니다. 문서/기획 단계에서는 구조 설계와 리포트 정리에 강한 도구를 사용하고, 구현 단계에서는 코드 작성과 구조 보조에 적합한 도구를 사용하며, 런타임 단계에서는 질문 생성과 답변 분석에 사용할 모델을 구분해 운영합니다.

### 각 AI 도구별 활용 전략을 작성해주세요.

문서 및 기획 단계의 AI 도구는 문제 정의를 선명하게 만들고, 대회 기준에 맞는 문서 구조를 정리하며, 리포트 항목별 초안을 준비하는 역할을 맡습니다. 구현 단계의 AI 도구는 폴더 구조 정리, 코드 골격 작성, 스키마 설계, 반복 작업 보조를 담당합니다. 런타임 AI는 제출물과 루브릭을 바탕으로 질문 3개를 생성하고, 학생 답변을 구조화된 기준으로 분석하며, 교사가 바로 읽을 수 있는 근거 중심 결과를 생성합니다.

이때 각 도구에는 역할을 분명히 나눕니다. 문서용 AI는 제품 메시지와 대회 정합성을, 구현용 AI는 코드 구조와 생산성을, 런타임 AI는 실제 제품 기능을 담당합니다. 또한 모든 런타임 출력은 JSON schema를 기준으로 검증하여 자유 텍스트 응답에 직접 의존하지 않도록 설계합니다.

### 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

모든 사례에 고성능 모델을 일괄 적용하지 않고, 질문 생성과 1차 판독에는 상대적으로 가벼운 모델을 사용하고, 불확실하거나 정밀 분석이 필요한 사례에만 고성능 모델을 적용하는 계층형 전략을 사용합니다. 또한 프롬프트 버전을 파일로 관리하고, 모델 버전과 실행 로그를 남기며, 질문 세트와 분석 리포트는 JSON schema 기반으로 구조화해 유지보수성과 재현성을 확보합니다.

샘플 데이터, 프롬프트 버전, 출력 스키마를 고정해 회귀 확인이 가능하도록 만들고, 교사 최종 판단 구조를 유지해 AI 자동 점수화 위험을 피합니다.
