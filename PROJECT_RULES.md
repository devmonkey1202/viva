# 프로젝트 시작 규칙

이 문서는 구현 전에 반드시 고정해야 하는 비가역 규칙을 정리한다. 아래 규칙은 개인 취향이 아니라 이 프로젝트와 공모전의 성격상 깨지면 안 되는 기준이다.

## 1. 제품 정체성 규칙

- 방향, 목표, 규약의 최상위 기준은 현재 폴더의 원본 문서들이다.
  `title.md`, `gongmo.md`, `① 개인정보 수집/이용 동의서.docx`, `② 참가 각서.docx`, `③ AI리포트.docx`
- 이후 구현, 문서, 리포트, 발표 자료는 위 원본에 적힌 대회 내용과 사용자 아이디어를 그대로 따른다.
- 원본과 다른 방향의 해석이나 확장은 사용자 승인 없이 추가하지 않는다.
- 제품의 핵심 문장은 언제나 다음과 같다.
  학생의 "진짜 이해"를 제출물 기반의 짧은 검증으로 증명하게 만드는 평가 검증 인프라.
- 제품 설명에서 `AI 탐지기`, `표절 탐지기`, `숙제 도우미`, `LMS 대체재`, `AI 튜터`라는 표현을 전면에 두지 않는다.
- 핵심 가치는 `교사의 검증 품질 고도화`, `학생의 이해 증명`, `운영자의 학습 품질 가시화`다.
- 최종 사용자 경험은 "감시"가 아니라 "공정한 이해 검증"으로 설계한다.

## 2. 제출 범위 규칙

- 이 프로젝트의 목표는 장난감 MVP가 아니라 `공모전 제출형 v1`이다.
- 내부 설계에서 MVP라는 표현을 써도, 외부 제출물은 핵심 시나리오에서 거의 완성형처럼 보여야 한다.
- 즉 범위는 좁게 잡되, 그 범위 안에서는 UI, 기능 흐름, 분석 결과, 문서, 배포가 모두 살아 있어야 한다.

- 첫 버전은 한 과목군과 한두 개 과제 유형에만 집중한다.
- 초기 지원 대상은 `개념/논리 검증이 분명한 서술형 과제 또는 프로젝트 설명문`이다.
- 첫 버전에서 반드시 포함할 기능은 아래와 같다.
  제출물 입력
  루브릭 입력
  질문 3개 생성
  학생 답변 수집
  이해 신뢰도와 근거 출력
  교사 최종 판단
  내보내기
- 첫 버전에서 거의 완성형으로 보여야 하는 품질 기준은 아래와 같다.
  실제로 동작하는 배포 URL
  끊기지 않는 핵심 happy path
  실패 시 재시도 또는 안내
  샘플 데이터 포함
  제출용 문서와 리포트 정합성
- 첫 버전에서 의도적으로 제외할 기능은 아래와 같다.
  전 과목 범용 지원
  완전 자동 채점
  영상 중심 면접 흐름
  LMS 완전 통합
  AI 작성 여부 점수화

## 3. AI 사용 규칙

- AI는 최종 판정자가 아니다. 교사가 최종 판정자다.
- AI 출력은 반드시 구조화된 스키마를 통과해야 한다.
- 질문 생성과 분석은 모두 프롬프트 버전, 모델 버전, 실행 시각, 입력 요약과 함께 남긴다.
- 키워드 매칭만으로 판정하지 않는다. 의미 수준 비교를 기본으로 한다.
- 불확실한 사례는 `불확실`로 남기고 교사 리뷰로 넘긴다.
- 자동 점수화는 금지한다.

## 4. 데이터 및 안전 규칙

- 공개 저장소에 실명, 전화번호, 메일, API 키, 토큰, 시크릿을 남기지 않는다.
- 데모 데이터는 합성 또는 익명화 데이터를 사용한다.
- 학생 검증 데이터는 최소 수집 원칙을 따른다.
- 영상보다 텍스트/음성을 우선한다.
- 질문, 답변, 근거, 누락 개념, 충돌 문장은 이의제기를 위해 저장 가능해야 한다.

## 5. 엔지니어링 규칙

- 비즈니스 규칙은 UI와 분리한다.
- 프롬프트는 코드와 분리해 버전 관리한다.
- 데이터 계약은 JSON schema로 명시한다.
- 도메인 용어를 우선한다. 임의 축약어를 남발하지 않는다.
- 함수는 한 책임만 가진다.
- 모듈은 입력과 출력을 분명히 드러낸다.
- 구현보다 문서가 앞선다. 다만 문서는 구현과 함께 즉시 갱신되어야 한다.

## 6. 대회 운영 규칙

- 최종 제출 기한은 `2026-04-13`이다.
- 내부 코드/문서 동결 시점은 그보다 앞당겨 잡는다.
- 최종 제출 후 커밋이 생기지 않도록 운영한다.
- 필수 제출물 5종이 모두 준비되기 전까지 제출 완료로 보지 않는다.
- AI 리포트에는 실제 사용한 도구와 실제 작업 방식을 기준으로만 작성한다.
- 제품 완성도는 코드만이 아니라 배포, 문서, 데모 흐름, 리포트까지 포함해 판단한다.

## 7. 의사결정 규칙

- 원본 문서에 이미 답이 있는 내용은 새로 재해석하지 않는다.
- 대회 기준과 사용자 아이디어가 충돌하지 않는 한 원문을 우선 보존한다.
- 범위를 넓히는 제안보다 검증 신뢰를 높이는 제안을 우선한다.
- 멋진 기능보다 심사 기준 적합성을 우선한다.
- "AI를 왜 써야 하는가"에 답하지 못하는 기능은 제외한다.
- 설명이 어려운 기능은 우선순위를 낮춘다.
- 빠른 데모보다 재현 가능한 데모를 우선한다.
