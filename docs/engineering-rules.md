# 엔지니어링 및 클린 코드 규칙

이 문서는 구현 단계에서 코드 품질을 지키기 위한 기준이다. 공모전 프로젝트라고 해서 구조를 희생하지 않는다. 다만 과도한 추상화 대신 실전적인 단순함을 유지한다.

## 1. 우선 원칙

- 코드보다 도메인 용어를 먼저 정한다.
- 작은 구조라도 계층 경계는 분명히 둔다.
- 한번 쓰고 버릴 코드처럼 작성하지 않는다.
- 다만 공모전 범위를 넘는 과설계는 피한다.

## 2. 설계 원칙

- 한 모듈은 한 책임만 가진다.
- UI는 도메인 규칙을 직접 계산하지 않는다.
- 도메인 규칙은 특정 프레임워크에 의존하지 않는다.
- 외부 API 호출은 래퍼 계층 뒤로 숨긴다.
- AI 출력은 항상 검증된 DTO로 변환한 뒤 사용한다.

## 3. 함수 작성 규칙

- 함수는 가능한 한 하나의 명확한 일을 해야 한다.
- 함수 이름만 보고 역할을 이해할 수 있어야 한다.
- 불리언 플래그로 여러 동작을 섞지 않는다.
- 반환값이 복잡해지면 구조체로 명시한다.
- 입출력 경계에서 유효성 검사를 수행한다.

## 4. 네이밍 규칙

- 축약어보다 도메인 이름을 쓴다.
- `data`, `info`, `util`, `manager`, `helper` 같은 모호한 이름은 피한다.
- 상태값은 명사형보다 의도가 드러나는 열거형을 사용한다.
- "왜형/전이형/반례형"처럼 도메인에 중요한 개념은 코드에서도 그대로 유지한다.

## 5. 파일 분리 규칙

- 컴포넌트 파일은 UI 책임만 가진다.
- 서비스 파일은 흐름 조합을 담당한다.
- 도메인 파일은 규칙과 상태를 담당한다.
- 스키마 파일은 검증 계약을 담당한다.
- 프롬프트 파일은 별도 저장한다.

## 6. 주석 규칙

- 주석은 이유를 설명할 때만 쓴다.
- 코드가 무엇을 하는지는 함수명과 구조로 드러나야 한다.
- 오래된 주석은 방치하지 않는다.
- TODO는 주인 없는 메모처럼 남기지 않는다. 조건과 맥락을 적는다.

## 7. 에러 처리 규칙

- 에러를 삼키지 않는다.
- 사용자 노출 메시지와 내부 로그 메시지를 분리한다.
- AI 응답 파싱 실패는 정상 흐름으로 취급하지 않는다.
- 재시도 가능한 에러와 불가능한 에러를 구분한다.
- 교사가 다시 실행할 수 있는 상태를 설계한다.

## 8. 테스트 규칙

최소 아래 테스트를 갖춘다.

- 질문 생성 출력 스키마 테스트
- 분석 출력 스키마 테스트
- 결과 분류 규칙 테스트
- 대표 시나리오 E2E 테스트
- 실패 시나리오 테스트

테스트 데이터는 아래를 포함한다.

- 이해 충분 사례
- 표면 암기 사례
- 제출물 의존 사례
- 핵심 오개념 사례
- 불확실 사례

## 9. 프롬프트와 코드의 분리

- 프롬프트는 코드 문자열로 장문 하드코딩하지 않는다.
- 프롬프트 파일은 버전명으로 관리한다.
- 프롬프트 변경은 코드 변경처럼 기록한다.
- 프롬프트 변경이 결과에 미치는 영향은 샘플 데이터로 확인한다.

## 10. 스키마 우선 원칙

- AI 출력은 자유 텍스트가 아니라 스키마를 기준으로 설계한다.
- 스키마가 바뀌면 저장 구조와 UI 영향을 같이 검토한다.
- 스키마는 문서와 코드 양쪽에 명확히 남긴다.

## 11. 로그 규칙

- request id 없이 AI 실행 로그를 남기지 않는다.
- prompt version과 model version 없는 로그는 불완전 로그로 본다.
- 개인정보가 포함된 raw payload는 최소한으로 저장한다.
- 분석 근거와 시스템 에러 로그를 분리한다.

## 12. 구성 관리 규칙

- 환경 변수는 예시 파일로 문서화한다.
- 비밀값은 저장소에 커밋하지 않는다.
- 개발/데모/제출 설정을 분리한다.
- 배포 환경 차이를 코드 안에 하드코딩하지 않는다.

## 13. UI 규칙

- 결과는 점수보다 근거 중심으로 보여준다.
- 교사와 학생 화면의 목적을 분리한다.
- 학생 화면은 감시 인상을 주는 문구를 피한다.
- 운영자 화면은 집계와 패턴 중심으로 구성한다.

## 14. 코드 리뷰 기준

아래 중 하나라도 설명이 약하면 수정한다.

- 이 모듈의 책임이 명확한가
- 도메인 용어가 일관적인가
- 실패 시 상태가 안전한가
- 프롬프트/스키마 변경 영향이 관리되는가
- 사용자에게 잘못된 확신을 주지 않는가

## 15. 금지 목록

- UI 컴포넌트 안에서 직접 모델 호출
- 타입 없이 AI 응답 문자열 그대로 사용
- 최종 점수 자동 확정
- 근거 없는 confidence 숫자 남발
- 전역 잡탕 유틸 폴더
- 의미 없는 추상 클래스와 인터페이스 남발
- 비밀키 하드코딩

## 16. 완료 정의

기능이 완료되었다고 말하려면 아래를 만족해야 한다.

- 문서 기준과 충돌하지 않는다.
- 스키마가 존재한다.
- 실패 상태가 정의되어 있다.
- 대표 테스트가 있다.
- 데모 흐름에서 설명 가능하다.
- 대회 메시지와 일치한다.
