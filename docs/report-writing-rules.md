# 리포트 및 대회 문서 작성 규칙

이 문서는 AI 리포트와 기타 공모전 제출 문서를 계속 작성할 때 따라야 하는 규칙을 정리한다. 목적은 `멋있어 보이는 문장`이 아니라 `심사 기준에 정확히 맞는 문장`을 만드는 것이다.

## 1. 기본 원칙

- `title.md`, `gongmo.md`, 공식 DOCX 3종에 적힌 내용과 사용자 아이디어를 그대로 따른다.
- 리포트 문장은 원본 기준을 다시 포장하는 것이지, 방향을 새로 바꾸는 수단이 아니다.
- 실제 구현된 내용만 쓴다.
- 과장하지 않는다.
- 모호한 슬로건보다 구체적인 흐름을 쓴다.
- 기술 설명과 제품 설명이 서로 어긋나면 안 된다.
- title.md의 제품 정체성을 벗어나지 않는다.
- 기록과 정리는 대회 대응의 핵심이므로 반드시 남긴다.
- 다만 같은 설명을 여러 문서에 반복해서 늘리지 않는다.

## 2. 반드시 유지할 제품 설명

- VIVA는 LMS가 아니다.
- VIVA는 AI 튜터가 아니다.
- VIVA는 AI 탐지기가 아니다.
- VIVA는 제출 이후 검증 레이어다.
- VIVA는 학생의 진짜 이해를 증명하게 만드는 평가 검증 인프라다.

## 3. 심사 기준별 서술 규칙

### 기술적 완성도

- 실제 화면 흐름
- 배포 상태
- 저장/분석/export 동작
- 실패 처리와 재시도

### AI 활용 능력 및 효율성

- 질문 생성과 분석에 AI가 왜 필요한지
- 어떤 모델을 어디에 배치했는지
- 비용 절감과 재현성 전략
- 구조화 출력과 스키마 검증

### 기획력 및 실무 접합성

- 교사 pain
- 학생 pain
- 운영자 pain
- 실제 교육 현장 도입 시 사용 맥락

### 창의성

- LMS/튜터/탐지기와 다른 관점
- AI 사용을 막는 대신 이해를 증명하게 하는 전환

## 4. AI 리포트 문장 작성 규칙

- "할 수 있다"와 "구현했다"를 구분한다.
- 구현한 것은 현재형으로 쓴다.
- 확장 계획은 미래형으로 쓴다.
- 아직 없는 기능을 현재형으로 쓰지 않는다.
- 자동 판정 과장 표현을 쓰지 않는다.

## 5. 절대 피할 표현

- 완벽하게 판정
- 완전 자동 평가
- 모든 과목에 즉시 적용
- 교사 업무를 없앰
- AI 작성 여부를 정확히 탐지

## 6. 권장 문서 세트

- README
- 문제 정의 문서
- 아키텍처 문서
- AI 전략 문서
- 샘플 입력/출력
- 제출 체크리스트
- AI 리포트 초안

## 7. 문서 정합성 체크

문서를 쓸 때마다 아래를 확인한다.

- README 설명과 리포트 설명이 같은가
- 리포트의 핵심 기능이 실제 화면과 같은가
- AI 도구 설명이 실제 사용 이력과 같은가
- 공모전 제출물 목록이 빠지지 않았는가
- 마감과 커밋 규칙을 문서에 반영했는가

## 8. 앞으로의 작성 원칙

- 새 기능을 만들면 관련 문서를 같이 갱신한다.
- 리포트는 마지막에 몰아서 쓰지 않고 구현과 같이 갱신한다.
- 발표 자료, README, AI 리포트는 같은 메시지를 유지한다.
