# 기술 스택 및 작업 방식 결정

이 문서는 VIVA를 어떤 언어와 스택으로 구현할지, 어떤 방식으로 클린 코드를 유지할지, 어떤 방식으로 작업 효율과 대회 대응 효율을 끌어올릴지 고정하는 문서다.

## 1. 개발 언어

주 언어는 `TypeScript`로 고정한다.

이유:

- 프론트엔드와 백엔드를 한 언어로 묶을 수 있다.
- 공모전 기간에 문맥 전환 비용이 적다.
- LLM 출력처럼 불안정한 데이터를 타입과 스키마로 강하게 통제하기 좋다.
- 문서, 스키마, API, UI를 한 흐름으로 관리하기 쉽다.

보조 언어:

- `SQL` 또는 ORM 스키마 정의
- 필요 시 아주 짧은 자동화용 `Python` 또는 셸 스크립트

원칙:

- 제품 핵심 로직은 TypeScript 안에 둔다.
- 보조 스크립트가 핵심 로직을 대신하지 않게 한다.

## 2. 권장 기술 스택

### 웹 애플리케이션

- `Next.js`
- `React`
- `TypeScript`

선택 이유:

- 교사용/학생용/운영자용 웹 UI를 빠르게 만들 수 있다.
- 라우팅, 서버 기능, 배포 흐름을 한 프로젝트에서 다루기 쉽다.
- 공모전 제출형 v1을 만들기에 속도와 완성도 균형이 좋다.

### 스타일링

- `Tailwind CSS` 또는 얇은 CSS 모듈 + CSS 변수

원칙:

- 디자인 시스템을 과하게 만들지 않는다.
- 대신 색상, 간격, 타이포, 상태 배지는 일관되게 관리한다.

### API / 서버 계층

- Next.js Route Handlers 또는 Server Actions

선택 이유:

- 초기 공모전 버전에서는 별도 서버를 완전히 분리하는 것보다 한 저장소 안에서 빠르게 완성형 흐름을 만드는 것이 유리하다.

### 데이터 검증

- `Zod`

선택 이유:

- 폼 입력, API 입출력, LLM 출력 검증을 하나의 방식으로 처리할 수 있다.
- 이 프로젝트는 AI 출력 검증이 중요하므로 스키마 우선 접근이 맞다.

### 데이터 저장

- 초기: `SQLite` 또는 간단한 관리형 DB
- 필요 시 확장: `PostgreSQL`

선택 이유:

- 초기 완성형 v1은 운영 규모보다 흐름 완결이 더 중요하다.
- 제출물, 질문 세트, 답변, 분석 리포트, 교사 판단 정도는 가볍게 저장해도 충분하다.

### ORM 또는 DB 레이어

- `Prisma` 또는 `Drizzle`

원칙:

- 복잡한 쿼리 최적화보다 명확한 모델링을 우선한다.
- 도메인 엔터티와 저장 구조가 쉽게 매핑되어야 한다.

### AI 호출 계층

- 모델 SDK 직접 호출
- 내부적으로는 `adapter` 패턴으로 분리

원칙:

- UI가 모델 SDK를 직접 호출하지 않는다.
- 질문 생성과 분석 호출 경로를 분리한다.
- 프롬프트 버전, 모델 버전, 실행 로그를 함께 저장한다.

## 3. 왜 이 스택이 이 프로젝트에 맞는가

이 프로젝트는 일반 CRUD보다 `LLM 출력 통제`, `짧은 공모전 기간`, `배포 완성도`, `문서 정합성`이 중요하다. 따라서 한 언어로 프론트/백/검증을 묶을 수 있고, 스키마 검증이 강하며, 빠르게 배포 가능한 TypeScript 중심 스택이 가장 실용적이다.

핵심은 "최신 기술 자랑"이 아니라 아래다.

- 빠르게 구현 가능
- 구조를 지키기 쉬움
- AI 출력 통제가 쉬움
- 문서와 코드가 잘 맞음
- 데모가 안정적임

## 4. 클린 코드 작성 방식

### 원칙 1. 도메인 우선

- 먼저 `과제`, `루브릭`, `제출물`, `질문 세트`, `답변`, `분석 리포트`, `교사 판단`을 중심으로 모델링한다.
- 화면이나 DB 테이블 이름보다 도메인 용어를 기준으로 코드를 쓴다.

### 원칙 2. 계층 분리

- `UI`
- `application/service`
- `domain`
- `ai`
- `storage`

이 다섯 계층을 섞지 않는다.

### 원칙 3. 스키마 우선

- AI 응답은 문자열로 직접 쓰지 않는다.
- 항상 스키마 검증 후 내부 객체로 변환한다.

### 원칙 4. 프롬프트 분리

- 프롬프트를 코드에 장문 문자열로 박아 넣지 않는다.
- `prompts/`에 버전 파일로 관리한다.

### 원칙 5. 한 책임 원칙

- 함수는 하나의 목적만 가진다.
- 컴포넌트는 화면 책임만 가진다.
- 도메인 규칙은 UI 바깥에 둔다.

### 원칙 6. 설명 가능한 결과

- 점수 하나를 뱉는 코드보다 근거 구조를 만드는 코드를 우선한다.
- 이 프로젝트의 출력은 교사가 읽어야 하므로 설명 가능성이 핵심이다.

## 5. 최고 효율을 뽑는 방식

효율은 두 종류로 나눠서 가져간다.

### 개발 효율

- 프론트/백을 TypeScript 하나로 통일
- 한 저장소에서 문서와 코드 함께 관리
- 스키마 먼저 작성 후 UI/API 동시 전개
- 샘플 데이터 기반 개발
- 리포트 초안과 구현을 동시에 갱신

### 런타임 효율

- 질문 생성은 상대적으로 가벼운 모델 우선
- 정밀 분석은 불확실 사례에만 사용
- 동일 입력 재분석은 캐시 고려
- 제출물 전문을 무조건 다 넣지 않고 필요한 구간만 사용
- 구조화 출력으로 후처리 비용 감소

### 공모전 효율

- 기능 구현과 리포트 작성을 분리하지 않음
- 화면 완성 시 바로 스크린샷/설명 문장 확보
- export 예시를 중간중간 저장
- `2026-04-12` 안에 내부 동결 목표
