# VIVA 디자인 블루프린트

VIVA의 디자인 기준을 하나의 문서로 고정한다. 이 문서는 `랜딩`, `teacher`, `student`, `operator`, `settings`를 같은 브랜드 안에서 서로 다른 목적의 화면으로 설계하기 위한 기준이다.

## 1. 목표

- `AI 서비스 광고`처럼 보이지 않는다.
- `교육용 검증 도구`처럼 보인다.
- `한국어 UI`에서 촌스럽지 않고, 과하지 않고, 정리된 느낌을 만든다.
- `랜딩`과 `실사용 화면`의 문법을 분리한다.
- `교사 판단 속도`와 `학생 응답 완료율`이 높아지는 방향으로 구조를 잡는다.

## 2. GDWEB 학습 기준

GDWEB 전시 작품 기준으로 아래 다섯 축만 참고한다.

- `실험적인`
- `깔끔한`
- `강렬한`
- `귀여운`
- `재미있는`

실제 참고한 대표 축은 다음과 같다.

- `실험적인`
  - 2024 World Webtoon Festival
  - VisualTrack
  - 동원그룹
  - 윤디자인그룹
- `깔끔한`
  - 해암문화재단
  - 현대 디에이치
  - INZENT ISS 2024
  - 팬엔터테인먼트
- `강렬한`
  - 현대제철
  - bold
  - JM COFFEE GROUP
- `귀여운`
  - 네이버 환경
  - 유성다과상
- `재미있는`
  - 2024 World Webtoon Festival
  - 네이버 환경
  - VisualTrack
  - 제일기획매거진

학습 결론은 단순하다.

- `랜딩`은 `실험적인 + 강렬한`을 제한적으로 흡수한다.
- `teacher / operator`는 `깔끔한`을 중심으로 간다.
- `student`는 `깔끔한`을 기반으로 `귀여운 / 재미있는`을 아주 약하게 섞는다.
- 화면 전체에 같은 톤을 깔지 않는다.

## 3. 브랜드 기준

### 3.1 로고

- 로고는 텍스트 로고만 사용한다.
- 기본 표기: `VIVA`
- 한글 보조 표기: `이해 검증 레이어`
- 로고는 장식하지 않는다.
- 두꺼운 기울임, 과한 테두리, 입체효과는 금지한다.

### 3.2 컬러

- 기본 배경: `#FFFFFF`
- 시그니처 블루: `#2B59FF`
- 본문 텍스트: `#111318`
- 보조 텍스트: `#5B6472`
- 보더: `#E7EBF0`
- 포인트 배경 블루: `#EEF3FF`
- 포인트 배경 민트: `#EAF7F1`
- 포인트 배경 피치: `#FFF1E8`
- 깊은 잉크 네이비: `#16213A`

컬러 운용 원칙:

- 흰색을 크게 쓴다.
- 블루는 적게 쓴다.
- 브랜드 배경은 면으로 쓰되, 넓은 gradient는 피한다.
- 강한 색은 `CTA`, `선택 상태`, `핵심 수치`, `핵심 키워드`에만 쓴다.

### 3.3 타이포

- 기본 서체: `Pretendard` 또는 동등한 한국어 산세리프
- 제목은 짧게, 강하게, 크게
- 설명은 1~2문장 이내
- 카드 설명은 1문장
- 줄 길이는 짧게 유지
- 과한 serif, 영문 display 중심 구성은 금지

권장 스케일:

- Display: `48-64`
- H1: `36-44`
- H2: `28-32`
- H3: `22-24`
- Body L: `18`
- Body: `16`
- Caption: `13-14`

### 3.4 간격과 면

- 기본 spacing scale: `8 / 12 / 16 / 24 / 32 / 48 / 64`
- radius: `12 / 18 / 24`
- shadow는 얕게, 최대 2단계만 사용
- 카드보다 `정렬`과 `그룹`이 먼저 보이게 설계한다

### 3.5 아이콘

- 둥근 단순 실무형
- 얇거나 과장되지 않은 선
- 픽토그램처럼 읽히는 수준
- 3D 아이콘, 유광 아이콘, 금융앱식 아이콘 남용 금지

### 3.6 비주얼 재료

- 추상 3D 오브젝트 허용
- 소프트 블러, 색면, 단색 오브젝트 허용
- 일러스트는 필수가 아니다
- 사진은 제품 신뢰를 높일 때만 제한적으로 사용
- 랜딩 외의 도구 화면은 비주얼보다 정보가 먼저 보여야 한다

### 3.7 모션

- 120ms ~ 220ms
- hover는 미세한 색 변화와 1px 수준 이동만
- loading은 짧고 조용하게
- 장식 목적의 motion 금지

## 4. 카피 기준

### 4.1 문장 규칙

- 설명은 짧게 쓴다.
- 한 문장에 한 의미만 넣는다.
- 기능 소개보다 결과를 먼저 말한다.
- 기술 자랑 대신 `질문 생성`, `근거 비교`, `교사 판단`, `오개념 요약`처럼 결과 중심으로 쓴다.

### 4.2 화면별 카피 톤

- 랜딩: 짧고 단정하게, 선언형
- teacher: 명확하고 실무형
- student: 부담 적고 쉬운 문장
- operator: 요약형, 분석형
- settings: 정책형, 설명형

### 4.3 금지

- 긴 기능 설명
- 의미 없는 감성 문구
- AI를 과장하는 표현
- 추상적인 카피 남발

## 5. 페이지 구조

### 5.1 공개 영역

- `/`
  - 제품 소개
  - 문제 정의
  - 사용자 역할별 진입
  - 검증 흐름 설명
- `/login`
  - 교사 / 운영자 로그인

### 5.2 교사 영역

- `/teacher`
  - 새 검증 세션 생성
  - 최근 세션 조회
  - 질문 생성
  - 학생 링크 관리
  - 분석 실행
  - 교사 판단 저장
- `/teacher/verifications/[verificationId]`
  - 세션 상세
  - 학생 응답 원문/STT 메타
  - 분석 근거
  - export
  - 링크 상태

### 5.3 학생 영역

- `/student/[verificationId]`
  - 링크 기반 진입
  - 질문 확인
  - 텍스트 / STT 답변
  - 제출 완료 상태

### 5.4 운영 영역

- `/operator`
  - 분류 분포
  - 누락 개념
  - 반복 오개념
  - 최근 세션
  - 운영 요약

### 5.5 관리 영역

- `/settings`
  - 계정
  - 검증 기본값
  - AI / 런타임
  - export / 알림
  - 개인정보 / 보관 정책

### 5.6 상태 페이지

- `/unauthorized`
- `/session-expired`
- `/not-found`

## 6. 핵심 사용자 흐름

### 6.1 교사 흐름

1. 로그인
2. `/teacher` 진입
3. 과제 / 루브릭 / 제출물 입력
4. 질문 생성
5. 학생 링크 발급
6. 학생 응답 대기
7. 응답 도착 후 결과 동기화
8. 분석 확인
9. 교사 판단 저장
10. 필요 시 세션 상세로 이동

이 흐름은 한 페이지에서 대부분 닫히고, 상세 검토가 필요할 때만 세션 상세로 내려간다.

### 6.2 학생 흐름

1. 링크 접속
2. 과제와 안내 확인
3. 질문 3개에 답변
4. 필요 시 음성 입력
5. 제출 완료

학생은 회원가입 없이 끝나야 한다.

### 6.3 운영자 흐름

1. 로그인
2. `/operator` 진입
3. 전체 분포 확인
4. 오개념 / 누락 개념 확인
5. 최근 세션 drill-down

## 7. 화면별 와이어 기준

### 7.1 랜딩

구조:

- 상단 바
  - `VIVA`
  - `교사용 워크벤치`
  - `학생 체험`
  - `운영 보기`
  - `로그인`
- Hero
  - 짧은 강한 제목
  - 한 줄 설명
  - 핵심 CTA 2개
  - 브랜드 비주얼 1개
- 문제/해결 섹션
  - `무엇이 문제인가`
  - `VIVA는 무엇을 하는가`
- 5단계 흐름
- 역할별 섹션
  - teacher
  - student
  - operator
- 하단 CTA

원칙:

- 전시형 인상을 주되, 정보 흐름은 명확해야 한다.
- 브랜드 장면은 강하지만 길게 끌지 않는다.

### 7.2 Teacher Workbench

구조:

- 상단
  - 제목
  - 현재 세션 상태
  - 주요 액션
- 좌측
  - 과제 설명
  - 루브릭
  - 제출물
- 우측 상단
  - 질문 세트
  - 학생 링크 카드
  - 세션 정책 카드
- 우측 하단
  - 분석 결과
  - 교사 판단
- 하단
  - 최근 세션 브라우저

원칙:

- 한 화면에서 주 흐름이 닫혀야 한다.
- 좌측은 입력, 우측은 결과로 고정한다.
- 카드가 아니라 워크벤치처럼 보여야 한다.

### 7.3 Teacher Detail

구조:

- 세션 메타
- 학생 응답 원문
- STT 메타
- 질문/답변 단위 근거
- 교사 판단 이력
- export

원칙:

- 읽기와 검토가 목적이다.
- 편집보다는 검증에 집중한다.

### 7.4 Student

구조:

- 상단 진행 상태
- 과제 요약
- 질문 카드 3개
- 답변 입력
- 음성 입력 안내
- 제출 CTA

원칙:

- mobile-first
- 문장과 단계 수를 줄인다.
- 불안감을 줄이고 완료율을 높인다.

### 7.5 Operator

구조:

- 상단 개요
- 핵심 수치
- 분류/판단 분포
- 누락 개념
- 반복 오개념
- 최근 세션 테이블

원칙:

- 예쁜 대시보드보다 빠르게 읽히는 운영 화면
- 카드 남용 대신 표와 요약을 섞는다

### 7.6 Settings

구조:

- 왼쪽 탭
  - 계정
  - 검증 기본값
  - AI / 런타임
  - export / 알림
  - 보안 / 개인정보
- 오른쪽 상세 패널

원칙:

- 설정은 실무형 관리 화면처럼 보여야 한다.
- 설명은 짧고, 기본값은 명확하게 보이게 한다.

## 8. 반응형 기준

### 8.1 랜딩

- mobile-first
- 큰 제목과 브랜드 장면은 모바일에서도 먼저 보이게
- 섹션은 1열 기준으로 시작하고 tablet 이상에서 2열 분기

### 8.2 Teacher / Operator

- desktop-first
- `>= 1200px`에서 메인 레이아웃 완전 전개
- `768px ~ 1199px`에서는 위아래 스택
- 모바일에서는 단순 보기와 핵심 액션만 유지

### 8.3 Student

- mobile-first
- 손가락 터치 기준 spacing
- 키보드가 떠도 CTA가 가려지지 않게

## 9. 구현 우선순위

1. 디자인 토큰 정리
2. 글로벌 레이아웃 / 헤더 재설계
3. 랜딩 재설계
4. teacher 워크벤치 재설계
5. student 모바일 플로우 재설계
6. operator 재설계
7. settings 재설계
8. 상태 페이지 정리
9. 반응형 / motion / empty / loading / error 마감

## 10. 체크리스트

- 랜딩과 툴 화면이 다른 문법으로 읽히는가
- teacher에서 입력 -> 질문 -> 링크 -> 분석 -> 판단 흐름이 끊기지 않는가
- student에서 모바일로 3문항 답변이 부담 없이 끝나는가
- operator에서 오개념과 누락 개념이 한눈에 읽히는가
- 화면 설명이 길지 않은가
- 시그니처 블루가 남용되지 않았는가
- 아이콘과 버튼과 배지가 같은 언어로 보이는가
- AI 광고처럼 보이지 않고 검증 도구처럼 보이는가
