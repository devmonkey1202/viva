# 의사결정 로그

중요한 제품, 구조, 운영 결정을 날짜 기준으로 기록한다.

## 2026-04-10

- 제품 정체성을 `제출 이후 이해 검증 레이어`로 고정했다.
  - 이유: 사용자 아이디어와 공모전 방향을 동시에 가장 정확하게 반영한다.

- 최종 판단의 책임은 교사에게 둔다고 고정했다.
  - 이유: 자동 점수화 서비스가 아니라 교사 판단 보조 구조가 제품 책임 구조에 맞다.

- 구현 범위를 `장난감 MVP`가 아니라 `공모전 제출형 v1`로 정의했다.
  - 이유: 코드, 배포, 문서, 리포트가 함께 완성돼야 한다.

## 2026-04-11

- 제품명을 `VIVA`로 확정했다.
  - 이유: 발표와 제품 설명에서 모두 읽기 쉽고, 검증 구조와의 의미도 맞다.

- AI 전략을 `파운데이션 모델 직접 학습`이 아니라 `VIVA 전용 검증 시스템 직접 구현`으로 정했다.
  - 이유: 일정, 구현성, 운영성, 리포트 설명 가능성을 모두 고려하면 이쪽이 더 강하다.

- 저장 구조를 `로컬 파일 + Neon/Postgres` 하이브리드로 정했다.
  - 이유: 로컬 개발 속도와 배포 안정성을 동시에 확보할 수 있다.

- 기본 배포 조합을 `Vercel + Neon`으로 정했다.
  - 이유: 무료 범위에서 시작하기 가장 현실적이다.

## 2026-04-13

- OpenAI Responses API를 기본 호출 구조로 정했다.
  - 이유: 구조화 응답 제어와 서버 측 운영이 쉽다.

- STT는 `Web Speech API + 텍스트 fallback`으로 정했다.
  - 이유: 무료 시연 가능성과 문서 정합성을 동시에 맞출 수 있다.

- 자동 검증을 `lint + unit test + build + HTTP smoke + CI` 조합으로 강화했다.
  - 이유: 수동 확인만으로는 반복 가능한 품질 증명이 어렵다.

- 학생 입력은 `링크 기반 진입`, 교사와 운영자는 `로그인 후 진입`으로 정했다.
  - 이유: 학생 진입 마찰은 줄이면서도 권한 경계를 유지할 수 있다.

- 설정은 헤더 드롭다운이 아니라 `/settings` 독립 페이지로 정했다.
  - 이유: 계정, AI, 보관 정책을 한 화면에서 명확히 다루기 쉽다.

- 교사 세션 상세는 `/teacher/verifications/[verificationId]`로 분리했다.
  - 이유: 워크벤치와 상세 검토는 목적이 다르다.

- 학생 링크는 잠금/재개방 상태를 가진다고 정했다.
  - 이유: 제출 이후 응답 수집을 통제하는 운영 기능이 필요하다.

- 디자인 기준 문서를 `docs/design-blueprint.md` 하나로 통합하기로 결정했다.
  - 이유: 디자인 시스템, 페이지 구조, 와이어, 카피 규칙이 여러 문서에 흩어지면 구현 단계에서 흐름이 다시 무너진다.

- 디자인 레퍼런스 학습 축을 `실험적인 / 깔끔한 / 강렬한 / 귀여운 / 재미있는`으로 고정했다.
  - 이유: GDWEB 작품을 무작정 넓게 보는 것보다, VIVA에 필요한 톤 축만 좁혀 보는 편이 더 실무적이다.

- 브랜드 기본값을 `흰 배경 + 시그니처 블루 + 텍스트 로고 + 둥근 실무형 아이콘`으로 고정했다.
  - 이유: VIVA는 검증 도구이므로 과한 연출보다 명확성과 신뢰가 먼저 보여야 한다.
