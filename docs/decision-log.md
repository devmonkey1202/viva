# 의사결정 로그

중요한 제품, 구조, 운영 결정을 날짜 기준으로 짧고 명확하게 남긴다.

## 2026-04-10

- 제품 정체성을 `제출 이후 이해 검증 레이어`로 고정했다.
  - 이유: 원본 아이디어와 공모전 방향을 동시에 가장 정확하게 반영한다.

- 최종 판단은 교사가 가진다고 고정했다.
  - 이유: AI 자동 판정 서비스가 아니라 교사 판단 보조 구조가 제품의 신뢰성과 책임 구조에 맞다.

- 구현 범위를 `장난감 MVP`가 아니라 `공모전 제출형 v1`로 정의했다.
  - 이유: 코드, 배포, 문서, 리포트, 제출 패키지가 함께 완성되어야 하기 때문이다.

## 2026-04-11

- 제품명을 `VIVA`로 확정했다.
  - 이유: 발표와 제품 설명에서 모두 사용하기 쉽고, 검증 구조와도 잘 맞는다.

- AI 전략을 `파운데이션 모델 직접 학습`이 아니라 `VIVA 전용 검증 시스템 직접 구현`으로 정했다.
  - 이유: 일정, 재현성, 운영성, 설명 가능성 기준에서 더 강하다.

- 저장 구조를 `로컬 파일 + Neon/Postgres` 하이브리드로 정했다.
  - 이유: 로컬 개발 속도와 배포 안정성을 동시에 확보하기 쉽다.

- 기본 배포 조합을 `Vercel + Neon(Postgres)`로 정했다.
  - 이유: 무료 범위에서 시작하기 가장 현실적이다.

## 2026-04-13

- OpenAI Responses API는 구조화 응답 중심으로 사용한다.
  - 이유: 자유 텍스트 파싱보다 실패 제어와 재현성이 높다.

- STT는 `Web Speech API + 텍스트 fallback`으로 구현한다.
  - 이유: 무료 데모 가능성과 문서 정합성을 동시에 맞출 수 있다.

- 자동 검증은 `lint + unit test + build + HTTP smoke + CI` 조합으로 강화한다.
  - 이유: 수동 확인만으로는 반복 가능한 품질을 증명할 수 없다.

- 교사 화면에는 학생 원문 답변과 전사 메타를 함께 보여준다.
  - 이유: 최종 판단 전에 실제 답변과 전사 품질을 교사가 직접 검토할 수 있어야 한다.

- 공개 페이지와 도구 페이지는 분리된 surface로 설계한다.
  - 이유: 소개용 장면과 실사용 워크벤치는 목적과 문법이 다르다.

- 인증 모델은 `교사/운영자 로그인`, `학생 링크 기반 진입`으로 간다.
  - 이유: 학생 진입 마찰을 줄이면서도 교사와 운영자의 권한 경계를 명확히 할 수 있다.

- 설정은 헤더 드롭다운이 아니라 `/settings` 독립 페이지에 둔다.
  - 이유: 역할, 정책, AI 런타임, 개인정보 보관 정책을 한 곳에서 관리해야 하기 때문이다.
