# 의사결정 로그

이 문서는 중요한 제품/기술 결정을 짧고 명확하게 남기는 로그입니다.

## 2026-04-10

- 제품 정체성을 `제출 이후 이해 검증 레이어`로 고정했다.
  - 이유: 단순 LMS나 AI 튜터보다 원본 아이디어와 공모전 방향에 직접 맞는다.

- 최종 판단은 반드시 교사가 가지는 구조로 고정했다.
  - 이유: 공정성과 설명 책임은 AI가 아니라 교사가 가져야 한다.

- 제출물 수준을 `가벼운 MVP`가 아니라 `공모전 제출형 v1`로 정의했다.
  - 이유: 이 대회는 구현, 배포, 문서, 리포트까지 함께 본다.

## 2026-04-11

- 제품명을 `VIVA`로 확정했다.
  - 이유: 발표와 심사에서 설명이 빠르고 제품명으로도 충분히 강하다.

- AI 전략을 `기반 모델 직접 학습`보다 `VIVA 전용 검증 시스템 직접 구현`으로 정했다.
  - 이유: 현재 중요 포인트는 재현 가능한 구조, 설명 가능한 결과, 운영 가능한 품질 관리다.

- 저장 구조를 `로컬 파일 + Neon/Postgres` 하이브리드로 정했다.
  - 이유: 로컬 개발 속도와 배포 안정성을 동시에 확보하기 위해서다.

- 배포 경로는 `Vercel + Neon(Postgres)`를 기본으로 정했다.
  - 이유: 무료 기준에서 가장 현실적인 조합이다.

## 2026-04-13

- OpenAI Responses API는 `json_object` 기반 구조화 응답으로 고정했다.
  - 이유: 자유 텍스트 파싱보다 실패 제어와 재현성이 높다.

- STT는 `Web Speech API + 텍스트 fallback`으로 구현했다.
  - 이유: 무료 시연 가능성과 원문 문서 정합성을 동시에 맞추기 위해서다.

- 자동 검증은 `lint + unit test + build + HTTP smoke + CI` 조합으로 강화했다.
  - 이유: 로컬에서만 되는 확인이 아니라, 반복 가능한 검증 흐름이 필요하다.

- 교사 화면에 학생 답변과 전사 메타 검토 영역을 추가했다.
  - 이유: 최종 판단 전에 실제 학생 답변을 직접 검토할 수 있어야 한다.

- 대형 UI 로직은 순수 함수와 서브컴포넌트로 분리하기 시작했다.
  - 이유: 기능이 동작하는 것만으로는 부족하고, 유지보수 가능한 구조가 필요하다.
