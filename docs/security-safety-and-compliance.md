# 보안, 안전장치, 규정 대응

이 프로젝트는 교육 솔루션이지만, 안전장치가 제품 신뢰의 핵심이다. 특히 공개 저장소 제출, 개인정보, 공정성, 오판 리스크를 함께 관리해야 한다.

## 1. 개인정보 원칙

- 데모에는 실제 학생 개인정보를 사용하지 않는다.
- 공개 저장소에는 실명, 연락처, 이메일, 계좌정보를 넣지 않는다.
- 샘플 데이터는 합성 또는 익명화 데이터를 사용한다.
- 학생 답변 원문은 필요한 범위만 저장한다.

## 2. 최소 수집 원칙

- 첫 버전은 영상보다 텍스트/음성을 우선한다.
- 얼굴, 표정, 시선 추적 같은 민감 정보는 수집하지 않는다.
- 학생 식별은 데모 기준 가명 ID로 충분하다.
- 과제 검증에 필요 없는 메타데이터는 저장하지 않는다.

## 3. 자동 판정 통제

- AI가 최종 점수를 부여하지 않는다.
- 최종 판단은 교사 승인 단계를 반드시 거친다.
- `불확실` 상태를 유지할 수 있어야 한다.
- 결과는 "참고 근거"로 제시하고 최종 결정권은 사람에게 둔다.

## 4. 이의제기 가능성 확보

학생 또는 교사가 결과를 검토할 수 있도록 아래를 남긴다.

- 생성된 질문
- 학생 답변
- 분석 근거 요약
- 누락 개념
- 충돌 문장
- 오개념 라벨
- 프롬프트 버전
- 모델 버전

## 5. 감시 인상 방지

- 제품 문구에서 `감시`, `의심`, `추적`, `적발` 같은 표현을 전면에 두지 않는다.
- 학생 화면에는 "이해를 확인하고 설명 기회를 제공한다"는 메시지를 사용한다.
- AI 사용 자체를 죄처럼 다루지 않는다.

## 6. 공정성 원칙

- 특정 표현 습관이나 화법만으로 이해 부족으로 판정하지 않는다.
- 답변 길이만으로 자동 불이익을 주지 않는다.
- 말투보다 개념 일치와 설명 구조를 우선 본다.
- 짧은 답변으로 충분히 설명된 경우를 허용한다.

## 7. 저작권 및 출품 규정 대응

공모전 참가 각서 기준으로 다음을 지킨다.

- 타인의 결과물을 무단 도용하지 않는다.
- 다른 공모전 제출작을 재활용하지 않는다.
- 외부 레퍼런스와 오픈소스는 출처와 라이선스를 점검한다.
- AI 생성 산출물도 최종 검토 책임은 팀에게 있다.

## 8. 공개 저장소 운영 원칙

- API 키는 `.env`로 분리하고 예시 파일만 남긴다.
- 테스트용 샘플에도 민감정보를 넣지 않는다.
- 제출 직전 비밀값 스캔을 수행한다.
- 발표 자료와 스크린샷에도 개인 정보가 노출되지 않게 한다.

## 9. 데이터 보존 원칙

MVP 기준 권장:

- 데모용 분석 데이터는 단기 보존
- raw AI 응답은 짧은 보존 또는 마스킹
- 집계 결과는 익명화 후 보존

공개 저장소 기준:

- 실제 사용자 데이터는 저장소에 포함하지 않는다.

## 10. 위험 시나리오와 대응

### AI가 오판하는 경우

- 결과를 확정이 아니라 근거 중심 보조로 표시한다.
- 교사 override를 허용한다.
- 불확실 상태를 유지한다.

### 학생이 과도한 부담을 느끼는 경우

- 답변 길이를 짧게 유지한다.
- 왜 이 검증이 필요한지 설명한다.
- 감시가 아니라 이해 증명이라는 메시지를 유지한다.

### 교사가 추가 시스템으로 느끼는 경우

- 입력 항목을 최소화한다.
- 루브릭 재사용 구조를 둔다.
- 결과를 바로 판단 가능한 형태로 보여준다.

### 공개 저장소에 민감정보가 남는 경우

- 커밋 전 점검 체크리스트를 운영한다.
- 제출 전 별도 점검을 수행한다.

## 11. 출시 전 필수 확인

- 자동 점수화 문구가 없는가
- 학생 화면에 감시 인상을 주는 표현이 없는가
- 샘플 데이터가 익명화되었는가
- 비밀키가 저장소에 없는가
- 결과 로그에 최소한의 설명 가능성이 있는가
