# AI 품질 및 평가 계획

이 문서는 AI 기능을 어떻게 검증할지 정리한다. 좋은 모델을 붙이는 것과 좋은 서비스를 만드는 것은 다르다. 이 프로젝트는 서비스 품질 기준으로 AI를 평가해야 한다.

## 1. 평가 철학

- 정답률 하나로 보지 않는다.
- 서비스 목적은 `이해 검증 보조`이므로 질문 품질, 근거 품질, 보수성, 설명 가능성을 같이 본다.
- 과하게 자신감 있는 오판을 가장 위험하게 본다.

## 2. 평가 대상

### 질문 생성 평가

- 질문이 제출물 기반인가
- 세 질문 축이 분리되는가
- 질문이 검증 기능을 하는가

### 분석 평가

- 기대 분류와 크게 어긋나지 않는가
- 근거가 설득력 있는가
- 누락 개념과 충돌 문장을 잘 짚는가

### 운영자 집계 평가

- 비슷한 오개념을 일관되게 묶는가
- 교사가 수업 개선에 쓸 수 있을 정도로 읽히는가

## 3. Golden Set 구성

필수 시나리오:

- case_01_understood
- case_02_surface_memorization
- case_03_submission_dependency
- case_04_core_misconception
- case_05_uncertain

각 케이스는 아래를 가진다.

- assignment
- rubric
- submission
- expected question intent
- student answers
- expected classification
- expected evidence points

## 4. 질문 생성 평가 기준

### 필수 통과 기준

- 왜형/전이형/반례형이 모두 존재
- 각 질문이 서로 다른 기능을 수행
- 질문이 제출물의 핵심 주장과 연결

### 감점 기준

- 템플릿 문구처럼 일반적임
- 제출물과 직접 연결되지 않음
- 세 질문이 사실상 같은 질문
- 루브릭 핵심 개념을 건드리지 않음

## 5. 분석 평가 기준

### 필수 통과 기준

- 분류 결과 존재
- confidence band 존재
- missing concepts 존재 가능
- contradiction findings 존재 가능
- teacher summary가 읽힘

### 질적 평가 질문

- 교사가 이 결과를 보고 판단할 수 있는가
- 왜 이런 결과가 나왔는지 설명되는가
- 지나치게 확신하지 않는가
- 학생 답변의 미묘한 차이를 반영하는가

## 6. 운영 품질 기준

- 스키마 실패율이 낮아야 한다.
- 재시도 이후에도 실패하면 안전하게 사용자에게 안내해야 한다.
- AI 실패가 전체 서비스 실패로 이어지면 안 된다.

## 7. 테스트 방식

### 오프라인 평가

- golden set으로 프롬프트 회귀 확인
- 스키마 검증 자동화
- 대표 케이스 수동 리뷰

### 온라인 평가

- 샘플 데이터 기반 실제 UI 흐름 점검
- 배포 환경에서 응답 시간과 실패율 점검

## 8. 합격선 방식

숫자 하나로 자르지 않고 아래를 동시에 본다.

- 질문 품질이 충분한가
- 분석 근거가 읽히는가
- 과도한 오판이 적은가
- 불확실 처리가 보수적인가
- 데모에서 신뢰를 주는가

## 9. 회귀 관리

- 프롬프트 버전이 바뀌면 golden set 재검증
- 스키마가 바뀌면 UI와 샘플 결과 재검증
- 모델 변경 시 대표 케이스 다시 비교

## 10. 사람 리뷰 규칙

수동 리뷰 시 아래를 기록한다.

- 맞음/애매함/틀림
- 어떤 근거가 좋았는지
- 어떤 근거가 부족했는지
- 질문이 더 날카로워져야 하는지
- 분류보다 explanation이 더 문제였는지

## 11. 가장 위험한 실패

- 표면 암기를 이해 충분으로 보는 경우
- 핵심 오개념을 놓치는 경우
- 근거 없는 confident output
- 일반론만 말하는 질문
- 불확실 사례를 무리하게 확정하는 경우

## 12. 출시 전 AI 품질 게이트

- 질문 생성 프롬프트 v1 고정
- 분석 프롬프트 v1 고정
- 질문 세트 스키마 통과
- 분석 리포트 스키마 통과
- golden set 기본 사례 검토 완료
- 샘플 데모 흐름 품질 확인
