# AI 개발 마스터 플랜

이 문서는 VIVA의 AI를 어떻게 설계, 구현, 검증, 운영할지 전체 그림을 고정한다. 이 프로젝트에서 AI는 부가 기능이 아니라 제품의 핵심 엔진이다. 따라서 모델 호출 한두 번으로 끝내지 않고 `역할 분리`, `스키마`, `품질 검증`, `비용`, `배포`, `안전장치`까지 같이 설계한다.

## 1. AI의 정확한 역할

이 제품에서 AI는 아래 역할을 맡는다.

- 학생 제출물과 루브릭을 읽고 학생별 질문 3개를 생성한다.
- 학생 답변을 제출물과 루브릭에 비추어 의미 수준에서 분석한다.
- 누락 개념, 오개념, 논리 충돌, 전이 실패를 구조화한다.
- 교사가 바로 읽을 수 있는 근거 중심 결과를 만든다.
- 여러 사례를 묶어 운영자용 오개념 패턴을 집계한다.

이 제품에서 AI는 아래 역할을 맡지 않는다.

- 최종 점수 확정
- 자동 합격/불합격 결정
- AI 작성 여부 탐지의 최종 판정
- 학생 감시 기능

## 2. AI 개발의 최상위 원칙

### 원칙 1. 제품 정체성 유지

- AI는 "이해 검증"을 위해 존재한다.
- "누가 썼는가"보다 "지금 이해하고 있는가"를 본다.

### 원칙 2. 역할 분리

- 질문 생성 AI
- 답변 분석 AI
- 집계/클러스터링 AI 또는 후처리

각 역할을 하나의 프롬프트로 섞지 않는다.

### 원칙 3. 구조화 출력

- 자유 문장 답변을 바로 서비스에 쓰지 않는다.
- 모든 AI 출력은 JSON schema를 통과해야 한다.

### 원칙 4. 보수적 판정

- 애매하면 `불확실`로 둔다.
- 억지로 확정하지 않는다.

### 원칙 5. 인간 최종 판단

- 교사가 최종 판단자다.
- AI는 판단 보조와 근거 구조화를 담당한다.

### 원칙 6. 배포 전제

- 로컬 데모용이 아니라 배포 가능한 서비스 구조를 전제로 개발한다.
- 타임아웃, 재시도, 실패 안내, 로그가 있어야 한다.

## 2-1. 직접 구현의 범위

VIVA는 `기반 모델을 처음부터 새로 학습하는 프로젝트`가 아니라 `이해 검증에 특화된 AI 시스템을 직접 설계하고 구현하는 프로젝트`로 간다.

직접 구현하는 대상:

- 제출물 맞춤 질문 생성 방식
- 이해 분석 기준과 분류 체계
- 규칙 기반 보조 검사
- JSON schema 기반 출력 계약
- 품질 평가용 golden set
- 비용, 재시도, 로그, 버전 관리 운영 체계

이번 제출 전략에서 중심에 두지 않는 것:

- 파운데이션 모델의 직접 학습
- 검증되지 않은 소량 데이터 기반의 급한 파인튜닝
- 모델 학습 자체를 과시하기 위한 연구형 확장

이렇게 결정한 이유:

- 현재 일정에서 가장 중요한 것은 배포 가능한 완성도다.
- 이해 여부 판단은 고품질 라벨 데이터가 없으면 학습 품질을 증명하기 어렵다.
- 공모전 심사 기준은 학습 여부보다 `기술적 완성도`, `AI 활용 능력 및 효율성`, `실무 접합성`에 더 직접적으로 연결된다.
- 서비스에 최적화된 AI 시스템 설계와 운영 증거가 오히려 더 강한 제출 근거가 된다.

## 3. AI 기능을 세 개의 엔진으로 분리한다

## 3-1. Question Generation Engine

목적:

- 제출물과 루브릭을 기반으로 학생별 검증 질문을 생성한다.

입력:

- 과제 설명
- 루브릭 핵심 개념
- 금지 포인트 또는 취약 포인트
- 학생 제출물

출력:

- 왜형 질문
- 전이형 질문
- 반례형 질문
- 각 질문의 의도
- 각 질문이 겨냥한 개념

성공 조건:

- 질문 3개가 서로 다른 역할을 가진다.
- 질문이 제출물과 연결되어 있다.
- 얕은 답변으로 빠져나가기 어렵다.

실패 조건:

- 너무 일반적인 질문
- 제출물과 무관한 질문
- 세 질문이 사실상 같은 질문인 경우

## 3-2. Understanding Analysis Engine

목적:

- 학생 답변이 실제 이해를 보여주는지 의미 수준에서 분석한다.

입력:

- 학생 제출물
- 루브릭
- 생성된 질문
- 학생 답변

분석 축:

- semantic alignment
- concept coverage
- transfer ability
- contradiction check

출력:

- 결과 분류
- confidence band
- 근거
- 빠진 개념
- 충돌 문장
- 오개념 라벨
- 재설명 포인트

성공 조건:

- 결과가 근거와 함께 제시된다.
- 교사가 읽었을 때 판정 이유가 이해된다.
- 점수보다 설명이 앞선다.

실패 조건:

- 근거 없이 결과만 제시
- 제출물과 답변의 실제 차이를 못 잡음
- 과도하게 단정적인 판정

## 3-3. Misconception Aggregation Engine

목적:

- 개인 결과를 모아 반/과정 차원의 반복 패턴을 본다.

입력:

- 여러 analysis report

출력:

- 반복 오개념 라벨
- 자주 빠지는 개념
- 단원별 취약 포인트

권장 방식:

- v1에서는 규칙 기반 집계와 간단한 라벨 정규화 우선
- 필요 시 LLM을 후처리로 사용

## 4. 실제 구현 구조

AI는 아래처럼 서버 계층 안에 둔다.

```text
UI
  -> API
    -> Application Service
      -> AI Service Adapter
        -> Prompt Loader
        -> Model Client
        -> Schema Validator
      -> Storage
```

핵심 규칙:

- UI는 모델 호출을 모른다.
- Application Service가 요청을 조합한다.
- AI Service Adapter는 오직 모델 호출과 검증만 담당한다.
- Schema Validator 통과 전에는 결과를 저장하지 않는다.

## 5. 모델 전략

모델명은 환경 변수로 관리하고 문서상 역할만 고정한다.

### fast_model

용도:

- 질문 생성
- 초안 요약
- 간단한 보조 판독

조건:

- 비용이 낮고 응답이 빨라야 한다.

### reasoning_model

용도:

- 의미 비교
- 논리 충돌 검토
- 불확실 사례 재분석

조건:

- 추론 안정성이 높아야 한다.

### 왜 역할 분리를 하는가

- 모든 요청을 고성능 모델에 태우면 비용과 대기시간이 커진다.
- 질문 생성과 정밀 분석은 요구 수준이 다르다.
- 운영 단계에서 비용 조절이 가능해진다.

## 6. 프롬프트 설계 전략

프롬프트는 세 종류를 기본으로 둔다.

- `question-generation.v1.md`
- `answer-analysis.v1.md`
- `misconception-clustering.v1.md`

각 프롬프트는 아래 공통 구조를 가진다.

- 역할 정의
- 입력 계약
- 판단 기준
- 금지 규칙
- 출력 스키마
- 보수적 판정 규칙

프롬프트 작성 원칙:

- 제품 설명과 판정 규칙을 분리해서 쓴다.
- 길게 설명하되 중복을 줄인다.
- 절대 자유 형식 출력을 허용하지 않는다.
- version을 파일명으로 관리한다.

## 7. 스키마 전략

v1 필수 스키마:

- `question-set.schema.json`
- `analysis-report.schema.json`
- `teacher-decision.schema.json`

추가 권장:

- `misconception-cluster.schema.json`
- `export-row.schema.json`

스키마 원칙:

- required 필드를 명확히 둔다.
- enum을 적극 사용한다.
- additionalProperties는 가능한 한 false로 둔다.
- UI에 필요한 필드와 저장에 필요한 필드를 같이 고려한다.

## 8. 데이터 흐름

### 질문 생성

1. 교사가 과제와 루브릭을 저장
2. 학생 제출물을 저장
3. 서버가 질문 생성 프롬프트 입력 구성
4. fast_model 호출
5. question-set schema 검증
6. 실패 시 재시도
7. 성공 시 저장

### 분석

1. 질문과 답변이 준비됨
2. 서버가 분석 입력 구성
3. reasoning_model 호출
4. analysis-report schema 검증
5. 실패 시 재시도 또는 `needs_retry`
6. 성공 시 결과 저장
7. 교사 화면에 결과 표시

### 집계

1. 여러 analysis report 조회
2. 오개념 라벨과 빠진 개념 정규화
3. 집계 결과 표시

## 9. 품질 확보 전략

품질은 세 레벨로 잡는다.

### 레벨 1. 계약 품질

- JSON 스키마 검증
- 타입 검증
- null/빈 응답 방지

### 레벨 2. 기능 품질

- 질문 3개가 역할상 분리되는가
- 제출물 기반 질문인가
- 분석 결과가 근거를 가지는가

### 레벨 3. 제품 품질

- 교사가 읽고 이해 가능한가
- 데모에서 설득력이 있는가
- 리포트 서술과 구현이 일치하는가

## 10. 품질 평가 방식

필수 golden set을 만든다.

- 이해 충분 사례
- 표면 암기 사례
- 제출물 의존 사례
- 핵심 오개념 사례
- 불확실 사례

각 사례마다 아래를 기록한다.

- 과제
- 루브릭
- 제출물
- 질문 기대 방향
- 학생 답변
- 기대 분류
- 기대 근거

이 golden set으로 프롬프트를 회귀 검증한다.

## 11. 비용 및 성능 계획

### 비용 절감

- 질문 생성은 fast_model 우선
- 정밀 분석은 reasoning_model
- 불확실 사례만 재분석
- 입력 전문을 무조건 다 넣지 않음
- 중복 요청 캐시 고려

### 응답 속도 목표

- 질문 생성: 사용자가 기다릴 수 있는 수준
- 분석: 데모에서 끊기지 않는 수준

v1에서는 절대적인 ms 숫자보다 "기다릴 수 있는 UX"가 중요하다.

## 12. 운영 로그와 관측성

각 AI 실행마다 아래를 남긴다.

- request id
- assignment id
- submission id
- prompt version
- model version
- elapsed time
- token usage
- validation status
- final status

로그 원칙:

- 개인정보는 최소화
- 디버깅 가능한 수준은 유지
- 결과 근거와 시스템 에러를 분리 저장

## 13. 실패 처리 전략

### 질문 생성 실패

- 재시도 1~2회
- 그래도 실패하면 사용자에게 재생성 버튼 제공
- 부분 응답은 버린다

### 분석 실패

- 재시도
- 실패 상태 저장
- 교사 재실행 허용

### 스키마 실패

- 저장 금지
- raw 응답을 내부 로그로만 남기고 사용자에게 노출 금지

## 14. 안전장치

- AI는 최종 판정자가 아니다.
- confidence는 band로만 드러낸다.
- 학생 감시 인상을 주는 문구를 피한다.
- 불확실 사례는 교사 검토로 넘긴다.
- 민감정보 입력을 최소화한다.

## 15. 배포 전제 개발 규칙

- API 키는 서버 환경 변수에만 둔다.
- 프론트엔드에서 직접 모델 호출 금지
- timeout 설정
- retry 정책 설정
- 실패 상태 UI 분리
- 샘플 데이터 준비
- 배포 후 실제 동작 로그 확인

## 16. 구현 우선순위

1. 스키마
2. 프롬프트
3. 샘플 데이터
4. AI 서비스 어댑터
5. 질문 생성 API
6. 분석 API
7. 결과 화면
8. 집계 최소 기능
9. 로그/관측성

## 17. 성공 기준

- 질문 생성이 제출물 기반으로 보인다.
- 분석 결과가 근거 중심이다.
- 불확실 사례를 안전하게 처리한다.
- 교사 최종 판단이 살아 있다.
- 배포 상태에서 AI 기능이 실제로 동작한다.
- AI 리포트와 구현 설명이 정확히 일치한다.
