# 파일 운영 규칙

이 문서는 파일 정리, 추가, 이동, 삭제를 어떻게 할지 정한다. 목적은 저장소가 커져도 구조가 무너지지 않게 하는 것이다.

## 1. 최상위 파일 원칙

루트에는 아래 성격의 파일만 둔다.

- 프로젝트 소개
- 핵심 규칙
- 작업 보드
- 원본 기준 문서
- 공통 설정 파일

추가 원칙:

- 문서 수를 늘리는 것 자체를 목표로 하지 않는다.
- 같은 역할의 문서가 겹치면 새 파일을 만들기보다 기존 문서에 통합한다.
- 다만 기준 문서, 실행 문서, 제출 초안처럼 역할이 명확히 다르면 분리한다.

## 2. 새 파일을 둘 위치

- 기준 문서: `docs/`
- 실제 제출 리포트 초안: `reports/`
- 프롬프트: `prompts/`
- 스키마: `schemas/`
- 샘플 데이터: `samples/`
- 데모 산출물: `artifacts/`
- 앱 코드: `apps/`
- 공통 패키지: `packages/`
- 보조 스크립트: `scripts/`
- 테스트: `tests/`

## 3. 원본 문서 처리 규칙

- `title.md`, `gongmo.md`, 공식 DOCX 3종은 원본으로 취급한다.
- 원본 파일은 이름을 임의로 바꾸지 않는다.
- 원본 파일 내용을 복사해 다시 쓰더라도, 기준은 항상 원본에 둔다.

## 4. 임시 파일 규칙

- 읽기용 임시 복사본은 작업 후 삭제한다.
- `final`, `real`, `new`, `new2`, `temp-final` 같은 이름을 쓰지 않는다.
- 필요하면 목적이 드러나는 이름을 쓴다.

## 5. 네이밍 규칙

- 문서: kebab-case
- 예시: `implementation-roadmap.md`
- 보고서 초안: 목적이 드러나는 이름
- 예시: `ai-report-draft.md`

## 6. 폴더 정리 규칙

- 비슷한 역할 파일을 루트에 흩뿌리지 않는다.
- 설명 문서는 `docs/` 또는 `reports/` 중 하나에 넣는다.
- 코드 없는 폴더라도 README로 용도를 남긴다.
- 중복 설명이 많은 문서는 주기적으로 합치거나 정리한다.
- 한 문서가 다른 문서를 거의 반복하면 둘 중 하나를 제거하거나 역할을 다시 정의한다.

## 7. 삭제 규칙

- 원본 문서는 삭제하지 않는다.
- 개인 정보가 섞인 임시 아티팩트는 즉시 삭제한다.
- 사용하지 않는 생성 산출물은 제출 직전 정리한다.

## 8. 공개 저장소 규칙

- 개인정보가 들어간 문서를 공개 저장소에 새로 복제하지 않는다.
- 비밀값이 포함된 설정 파일은 커밋하지 않는다.
- 스크린샷에도 민감정보가 없는지 확인한다.

## 9. 파일 추가 전 질문

- 이 파일은 기존 파일과 역할이 다른가
- 이 파일은 제출물 완성도를 높이는가
- 이 파일은 이후에도 찾기 쉬운 위치에 놓이는가
- 이 내용은 기존 문서에 합칠 수 없는가

아니면 새 파일을 만들지 않는다.
