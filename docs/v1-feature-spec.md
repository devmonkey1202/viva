# VIVA v1 기능 명세

이 문서는 현재 구현 중인 VIVA의 제품 범위, 역할별 흐름, 페이지 구조, 권한 모델, 설정 구조를 한 번에 고정하기 위한 기준 문서다.

## 1. 제품 정의

- VIVA는 LMS가 아니다.
- VIVA는 AI 튜터가 아니다.
- VIVA는 AI 탐지기가 아니다.
- VIVA는 제출 이후 학생이 실제로 이해했는지 검증하는 평가 검증 레이어다.

핵심 흐름은 아래와 같다.

1. 교사가 과제, 루브릭, 학생 제출물을 정리한다.
2. AI가 학생 제출물 기반 검증 질문 3개를 만든다.
3. 학생이 텍스트 또는 음성으로 짧게 답한다.
4. 시스템이 제출물, 질문, 답변, 루브릭을 비교 분석한다.
5. 교사가 근거를 보고 최종 판단한다.
6. 운영자가 반복 오개념과 패턴을 본다.

## 2. 역할과 우선 사용자

### 교사

- 과제와 루브릭 입력
- 학생 제출물 등록
- 질문 생성
- 학생 답변 검토
- AI 분석 근거 검토
- 최종 판단
- 결과 export

### 학생

- 질문 확인
- 답변 작성 또는 음성 전사
- 제출 완료 확인

### 운영자

- 분류 분포 확인
- 반복 오개념 확인
- 누락 개념 패턴 확인
- 최근 세션 확인

## 3. v1에서 반드시 제공해야 하는 사용자 흐름

### 교사 흐름

1. 새 검증 세션 시작
2. 과제 요약 입력
3. 루브릭과 핵심 개념 입력
4. 학생 제출물 입력
5. 질문 생성
6. 학생 링크 복사 또는 공유
7. 학생 답변 수집
8. 분석 결과 확인
9. 교사 최종 판단 저장
10. JSON/CSV export
11. 최근 세션 다시 열기

### 학생 흐름

1. 링크 접속
2. 세션 요약 확인
3. 질문 3개 확인
4. 텍스트 또는 음성으로 답변
5. 제출 완료 상태 확인

### 운영자 흐름

1. 전체 결과 개요 확인
2. AI 분류 분포 확인
3. 교사 판단 분포 확인
4. 누락 개념과 오개념 패턴 확인
5. 최근 세션 목록 확인

## 4. 페이지 구조

페이지는 `소개용 surface`, `도구용 surface`, `링크 기반 student surface`, `설정/운영 surface`를 분리해서 가져간다.

### 공개 페이지

- `/`
  - 제품 소개
  - 문제 정의
  - 제품 원칙
  - 역할별 진입
  - 데모 진입 안내

공개 영역은 최대한 얇게 유지한다. 소개 페이지를 여러 장으로 늘리는 것보다 하나의 밀도 높은 메인 페이지로 정리하는 편이 맞다.

### 인증 필요 페이지

- `/teacher`
  - 교사 메인 워크벤치
  - 새 세션 생성
  - 최근 세션 검색과 재오픈
  - 분석 결과 검토
  - 최종 판단 저장

- `/teacher/verifications/[verificationId]`
  - 세션 상세 보기
  - 공유 링크 확인
  - 질문 세트 보기
  - 답변 검토
  - 분석 및 최종 판단 검토
  - export

- `/operator`
  - 운영자 요약 대시보드
  - 전체 분포, 오개념, 누락 개념, 최근 세션

- `/settings`
  - 설정 허브
  - 계정, 워크스페이스, AI/런타임, 알림/내보내기, 보안 항목을 탭으로 분리

### 링크 기반 페이지

- `/student/[verificationId]`
  - 학생 전용 답변 페이지
  - 로그인 없이 링크 기반 진입을 우선 지원
  - 필요 시 이름 또는 확인 코드 입력을 추가할 수 있다

### 시스템 페이지

- `/404`
  - 잘못된 경로 안내

- 세션 만료 상태
  - student 링크가 만료되었거나 닫혔을 때 안내

- 권한 없음 상태
  - teacher/operator가 권한 없는 페이지에 접근했을 때 안내

## 5. 인증과 권한 모델

v1의 권한 모델은 아래처럼 단순하게 간다.

### 교사 / 운영자

- 로그인 필요
- 교사는 자신이 만든 세션을 조회하고 판단할 수 있어야 한다
- 운영자는 요약 페이지와 조직 단위 결과를 볼 수 있어야 한다

### 학생

- 별도 회원가입 없이 링크 기반 진입을 기본값으로 둔다
- 링크 기반 진입이므로 아래 안전장치를 둔다
  - 세션 상태 확인
  - 세션 만료 처리
  - 제출 완료 후 재제출 정책 명시

### 로그인 / 로그아웃

- 공개 페이지 헤더에는 `로그인`만 노출
- 인증된 교사/운영자 surface에는 우측 상단 사용자 메뉴를 둔다
- 사용자 메뉴에는 아래를 넣는다
  - 내 계정
  - 설정
  - 로그아웃

v1에서는 `교사/운영자 로그인 필요`, `학생은 링크 기반`이 가장 자연스럽다. 학생 로그인까지 강제하면 진입 마찰이 커지고, 현재 제품의 핵심 가치와도 맞지 않는다.

## 6. 설정 구조

설정은 별도 페이지로 빼고, 한 화면 안에서 탭 구조로 정리한다. 현재 제품 성격상 설정을 헤더 드롭다운 안에 억지로 숨기는 것보다 `/settings` 한 곳에 모으는 것이 맞다.

### 계정 설정

- 이름
- 소속
- 역할
- 기본 언어

### 검증 기본값

- 기본 질문 길이 가이드
- 학생 답변 시간 제한 여부
- 음성 답변 허용 여부
- 텍스트만 허용 여부
- 재분석 기본 옵션

### AI / 런타임 설정

- 질문 생성 모델
- 분석 모델
- fallback 허용 정책
- STT 사용 정책
- 로그 저장 범위

### export / 알림 설정

- 기본 export 형식
- 요약 다운로드 옵션
- 제출 완료 알림 정책

### 보안 / 개인정보 설정

- 세션 보관 기간
- 원문 답변 저장 여부
- 전사본 저장 여부
- 민감 정보 마스킹 정책

## 7. 페이지별 필수 구성 요소

### 메인 페이지 `/`

- 한 줄 제품 정의
- 무엇이 아닌가
- 핵심 흐름
- 역할별 surface 안내
- teacher 진입 CTA
- operator 진입 CTA

### 교사 워크벤치 `/teacher`

- 상단 상태 영역
- 과제/루브릭 입력 영역
- 제출물 입력 영역
- 질문 세트 생성 영역
- 학생 링크 영역
- 학생 답변 검토 영역
- 분석 근거 영역
- 최종 판단 영역
- 최근 세션 브라우저

### 세션 상세 `/teacher/verifications/[verificationId]`

- 세션 메타데이터
- 질문 세트
- 학생 원문 답변
- 음성 전사 메타
- 분석 결과
- 최종 판단
- export

### 운영자 요약 `/operator`

- 전체 메트릭
- AI 분류 분포
- 교사 판단 분포
- 누락 개념 상위 목록
- 오개념 상위 목록
- 최근 세션 목록

### 학생 페이지 `/student/[verificationId]`

- 과제 요약
- 질문 카드 3개
- 텍스트 답변
- 음성 답변 버튼
- 전사 상태 안내
- 제출 상태
- 오류 및 만료 상태

## 8. 필수 편의 기능

기능 자체뿐 아니라 실사용 편의성을 위해 아래 항목이 필요하다.

- 최근 세션 검색
- 세션 재오픈
- student 링크 복사
- STT 미지원 브라우저 fallback
- 저장 상태 표시
- AI fallback 상태 표시
- export 다운로드
- 빈 상태 화면
- 실패 상태 화면
- 제출 완료 상태 화면

## 9. 필수 상태 설계

각 페이지는 정상 상태만 있으면 안 된다. 최소한 아래 상태를 가져야 한다.

- loading
- saving
- analyzing
- saved
- empty
- error
- disabled
- expired
- unauthorized

## 12. 구현 원칙

- 소개 페이지와 도구 페이지 문법을 분리한다
- teacher/operator는 desktop-first로 설계한다
- student는 mobile-first로 설계한다
- AI는 결과를 내리는 척하는 장식이 아니라 질문 생성과 근거 구조화 UX로 드러나야 한다
- 설정은 적게 보이되, 역할과 정책을 통제할 수 있을 만큼은 있어야 한다
- 페이지를 줄이기 위해 역할을 섞지 않는다
